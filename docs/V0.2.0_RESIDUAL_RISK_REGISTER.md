## R-WS13.6-01 - Contact Duplication During Peer Discovery (2026-03-14 Audit)

- Phase: WS13.6 (Contact Persistence)
- Status: Open
- Severity: MEDIUM
- Impact: Database duplication, contact merge complexity, potential UI display issues
- Evidence:
  - 2026-03-14 fresh-install contact audit revealed duplicate `onPeerIdentified` callbacks for same peer ID
  - MeshRepository logs show: auto-create at 18:22:49.396, then duplicate create at 18:22:52.530 (4 seconds later)
  - Same peer ID (`93a35a87...`) appeared twice despite single discovery event
  - Relay peer (external Internet relay) was auto-discovered and auto-created as contact
- Root Cause:
  - MeshEventBus or discovery stack may be emitting duplicate IdentityDiscovered events
  - Contact creation callback not idempotent (no "already exists" check)
  - No deduplication logic on peer promotion callback
  - Database may support duplicate peer_ids (if insert-based, not upsert-based)
- Fix Required:
  - Verify `onPeerIdentified` callback deduplication in MeshRepository
  - Implement idempotent contact upsert (not insert)
  - Add unique constraint on peer_id in contacts table
  - Create migration to clean existing duplicates
  - Test: Fresh install + relay discovery should create 1 contact, not 2
- Audit Location: `tmp/scm_audit_logs/contact_audit_2026-03-14/CONTACT_PERSISTENCE_AUDIT_2026-03-14.md`
- Next Action: Design fix + implement idempotent contact creation before v0.2.1 release

## R-WS12.44-01 - 5-node visibility can still be partially indeterminate when local own-ID emission is missing from the current log window

- Phase: WS12.44
- Status: Open
- Impact: Operators can under-report true mesh visibility if they treat missing local-own-ID capture as proof of disconnection, especially on physical iOS-device and pre-existing headless relay runs.
- Evidence:
  - Pre-WS12.44 `run5.sh` mixed physical iOS app console output with host/system Bluetooth logs in a single `ios-device.log`, causing app-level peer-ID capture and host BLE radio activity to be conflated.
  - The 2026-03-13 `logs/5mesh/20260313_034519` artifact set showed strong Android BLE delivery evidence (`Identity beacon from`, `Characteristic write successful`, `delivery_attempt ... medium=ble outcome=accepted`) while the old matrix still reported only `25% connected` and `ios_dev own id = unknown`.
  - Headless own-ID capture could also be misreported because the old parser fell back to peer-frequency heuristics and remote relay-agent strings instead of requiring local-own-ID evidence.
  - WS12.44 mitigation now in place:
    - `run5.sh` splits `ios-device.log` (app console) from `ios-device-system.log` (host/system Bluetooth + Multipeer context),
    - pre-existing physical iOS sessions are no longer relaunched just to obtain app console output,
    - the visibility matrix now counts only known own IDs and marks missing own IDs as collector gaps,
    - duplicate/ambiguous own-ID inference is suppressed,
    - GCP log capture now prepends a recent `docker logs --tail 200` snapshot before incremental polling.
- Remaining ambiguity:
  - Physical iOS device runs still cannot passively guarantee local-own-ID capture without a fresh app startup line in the current window.
  - Pre-existing GCP/OSX relay sessions can still miss own-ID lines in short runs when startup context is outside the capture window.
  - Android BLE fallback telemetry can still report the requested fallback MAC in `delivery_attempt ... medium=ble outcome=accepted` while the authoritative `BleGattClient` callback success is emitted for the fresher connected GATT address actually used.
- Next action:
  - Use the upgraded `run5.sh` output as the authoritative interpretation surface for future 5-node runs.
  - Treat unknown own IDs as collector gaps unless transport evidence is also empty.
  - If deterministic passive own-ID capture becomes a hard release gate, add an explicit local-own-ID export/status surface rather than reintroducing heuristic inference.
