# SCMessenger v0.2.0: API 35 Regression Test Plan

Author: Antigravity
Status: Active
Last Updated: 2026-03-08

## Context

Google Play now requires all new or updated apps to target Android API Level 34 at minimum, and strongly encourages targeting 35 (Android 15) to maintain store visibility. We have bumped `compileSdk` and `targetSdk` to 35 to satisfy these requirements for the initial Google Play Internal Testing release.

This document outlines the testing surface area to ensure no behavior regressions result from complying with Android 15 policies.

---

## 1. Foreground Services & Background Execution (Critical)

### The Risk

Android 15 introduces stricter enforcements on when services can be launched from the background, and strictly limits how long background tasks can run without explicit user interactions. Because SCMessenger runs a background BLE mesh and mDNS listener, this is the most likely failure point.

### Test Scenarios

1. **Background BLE Discovery:**
   - Terminate the app entirely on the Android device.
   - Wait 5+ minutes to allow the OS to put the app in a deeper background state.
   - Send a message from an iOS device over BLE with Wi-Fi disabled.
   - **Expected Result:** Android wakes via BLE notification, processes the incoming message, and displays a Push Notification, without crashing due to an illegal foreground service start.
2. **Wi-Fi Aware / mDNS Multicast:**
   - Place the Android application in the background (Screen off).
   - Attempt a local Wi-Fi peer discovery from another device.
   - **Expected Result:** The `NSDManager` local network listener remains active or correctly re-registers without throwing a `SecurityException` unique to Android 15.

---

## 2. Notification Trampolines & Intents

### The Risk

Android 15 tightly locks down clicking a notification that triggers an intermediary broadcast receiver or service instead of directly starting an Activity (known as a notification trampoline).

### Test Scenarios

1. **Direct DM Notification Click:**
   - Wait for a Direct Message via BLE while the app is backgrounded.
   - Tap the system notification payload when it arrives.
   - **Expected Result:** The Intent directly resolves to the `MainActivity` (or specific chat Compose route) instantly, rather than silently failing to launch the UI.

---

## 3. Storage and Data Sandboxing

### The Risk

Targeting API 35 enforces new granular media and local storage restrictions. Because we rely on `java.io.File` via sled/Rust for storing the entire encrypted message history, any stealth OS path restrictions will break the node initialization.

### Test Scenarios

1. **Database Initialization (Cold Start):**
   - Clear app cache (not data). Start the app fresh.
   - **Expected Result:** Sled database initializes properly in `/data/user/0/com.scmessenger.android/files` without permission denials.
2. **Identity Backup (File Export):**
   - _If applicable during alpha:_ Export the keypair file to external storage using the `MediaStore` APIs or Scoped Storage selectors.
   - **Expected Result:** File saves successfully without triggering deprecated `WRITE_EXTERNAL_STORAGE` crashes.

---

## 4. UI Rendering: Edge-to-Edge Mandatory

### The Risk

On Android 15 (API 35), Google forces apps into an "edge-to-edge" layout by default. Status bars and navigation bars become transparent, and content draws underneath them. Since we use Jetpack Compose, some UI elements might be clipped under the OS nav bar.

### Test Scenarios

1. **Bottom Navigation & Chat Input:**
   - Open a chat thread.
   - Verify the text input field is not positioned _underneath_ the Android gesture bar or system navigation buttons.
2. **Top App Bar:**
   - Verify the top status indicators (Relay status, BLE status) do not clip underneath the camera cutout or OS clock text.
   - **Fix if discovered:** Apply `Modifier.systemBarsPadding()` or `Modifier.navigationBarsPadding()` to the root Compose `Scaffold`.

---

## 5. Summary / Go-Forward Policy

Run these four test suites internally on a physical device running Android 14+ or an Android 15 Emulator. If no crashes are observed in logcat, the API 35 bump is verified stable for the active beta cohort.
