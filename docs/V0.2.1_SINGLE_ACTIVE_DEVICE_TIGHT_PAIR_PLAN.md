# SCMessenger Single Active Device (Tight Pair) Plan

Status: Planned, not started
Decision: Target `v0.2.1` (deferred from `v0.2.0`)
Date: 2026-03-02
Phase anchor: `WS13` (v0.2.1 kickoff)

---

## 1) Decision summary

The full "Single Active Device per Identity" feature is deferred to `v0.2.1`.

Rationale:

1. It is a cross-cutting net-new protocol/state feature, not a completion of already-started `v0.2.0` scope.
2. It touches identity, contacts, transport protocol, relay custody state machine, UI behavior, and migration.
3. Current `v0.2.0` focus is reliability hardening + parity + app-store alpha readiness; introducing this now materially increases regression risk.

This plan keeps the feature fully specified so it can be executed directly after `v0.2.0` release-gate closure.

Tracking label: this feature is executed as `WS13`, decomposed into `WS13.1` through `WS13.6` below.

---

## 2) Product objective

Enforce a strict destination pairing:

`(identity_public_key, active_device_id)`

to prevent:

1. multi-device active collisions,
2. stale/recycled identity routing,
3. ambiguous relay custody ownership.

---

## 3) Core model (tight pair)

1. `identity_id`: Ed25519 public key (portable, seed-derived).
2. `device_id`: CSPRNG UUIDv4 generated per installation.
3. `seniority_timestamp`: activation timestamp for this installation instance.

Constraints:

1. `device_id` is generated once at install/identity restore and persisted.
2. Registration assertions are cryptographically bound to identity ownership (signature).
3. Relay routing/custody decisions are made against the active `(identity_id, device_id)` state.

## 5) State machine and policy rules

### 5.1 Registration collision handling

When new device registers for an identity with existing `Active`:

1. Reject with conflict by default.
2. Allow takeover only if existing `last_seen` is older than stale threshold (`14 days`).
3. On stale takeover, purge old-device pending custody entries, then accept new active device.

### 5.2 Deregistration handling

1. Clean handover (`target_device_id` present):
   - set `Handover(target_device_id)`,
   - migrate pending custody target from old device to target device.
2. Identity abandonment (no target):
   - set `Abandoned`,
   - purge pending custody for that identity.

### 5.3 Custody acceptance behavior

Given inbound relay request with destination identity + intended device:

1. `Active(match)` -> accept.
2. `Active(mismatch)` -> reject with recycled/invalid identity status.
3. `Handover(target)` -> accept and redirect/store against target device.
4. `Abandoned` -> reject with recycled/invalid identity status.
5. Legacy/no intended device:
   - compatibility mode during migration, then tighten in enforcement phase.

---

## 6) Security and correctness requirements

1. All registration/deregistration requests must be signed by identity private key.
2. Enforce payload canonical serialization before verify/sign.
3. Add anti-replay controls (timestamp validity window and/or monotonic server-side observed sequence).
4. Reject malformed UUIDs and invalid identity format early.
5. Audit-log registry transitions for forensic traceability.

---

## 7) Backward compatibility and rollout strategy

Recommended rollout for `v0.2.1`:

1. Phase A (compat mode):
   - old clients can still send without `intended_device_id`,
   - relay records and enriches route metadata where available.
2. Phase B (enforced mode):
   - strict active-device enforcement for clients advertising support.
3. Phase C (full policy):
   - tighten legacy path once client upgrade penetration is sufficient.

This avoids immediate breakage of existing `v0.2.0` clients.

  - registration/deregistration payloads serialize canonically,
  - signatures validate against identity ownership,
  - malformed UUID/identity/signature inputs fail closed without mutating relay state.
- **2026-03-11 implementation status: COMPLETE (Rust transport layer)**
  - `IronCoreBehaviour` now exposes `/sc/registration/1.0.0` as an additive request/response protocol with `RegistrationMessage::{Register,Deregister}` wrapper messages.
