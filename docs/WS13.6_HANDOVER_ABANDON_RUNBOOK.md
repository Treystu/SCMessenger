# WS13.6 Handover & Abandon Runbook

Status: Active  
Last updated: 2026-03-14  
Phase: WS13.6 (final WS13 workstream)

---

## 1. Overview

This runbook documents the operational procedures for device handover and identity abandonment flows in the WS13 tight-pair system. These flows handle the lifecycle of device identity transitions.

---

## 2. Handover Flow (Clean Device Transition)

### 2.1 Scenario

User is migrating from Device A to Device B (e.g., new phone upgrade). Device A is still active and can initiate a clean handover.

### 2.2 Procedure

#### Step 1: Device B Registers

Device B generates a new `device_id` (UUIDv4) and sends a registration request:

```
RegistrationPayload {
    identity_id: <same identity as Device A>,
    device_id: <new UUIDv4>,
    seniority_ts: <current timestamp>
}
Signature: Ed25519 signature over canonical payload
```

**Expected outcome**: Registration is rejected with conflict (Device A is still active).

#### Step 2: Device A Initiates Handover

Device A sends a deregistration request with target device:

```
DeregistrationPayload {
    identity_id: <identity>,
    from_device_id: <Device A's device_id>,
    target_device_id: <Device B's device_id>
}
Signature: Ed25519 signature over canonical payload
```

**Expected outcome**:
1. Registry state transitions to `Handover(target_device_id=Device B)`
2. Pending custody records are retargeted from Device A to Device B
3. Device A is no longer the active device

#### Step 3: Device B Re-Registers

Device B sends the registration request again.

**Expected outcome**:
1. Registry state transitions to `Active(device_id=Device B)`
2. Device B is now the active device
3. Pending custody records are delivered to Device B

### 2.3 Verification

```bash
# Check registration state
curl -X GET http://localhost:8080/api/v1/registration/state/<identity_id>

# Expected response after handover:
{
    "state": "Active",
    "device_id": "<Device B's device_id>",
    "last_seen": <timestamp>
}
```

### 2.4 Failure Modes

| Failure | Cause | Recovery |
|---------|-------|----------|
| Handover stuck in `Handover` state | Device B never re-registers | Device B must complete registration; or stale takeover after 14 days |
| Pending records orphaned | Handover completed but records not retargeted | Check relay custody store; manual retarget may be needed |
| Signature mismatch | Wrong identity key used | Verify Device A is using correct identity private key |

## 3. Abandonment Flow (Lost/Stolen Device)

### 3.1 Scenario

User has lost access to Device A (lost, stolen, or broken) and needs to abandon it without a clean handover.

### 3.2 Procedure

#### Step 1: Device B Registers (Conflict)

Device B sends a registration request.

**Expected outcome**: Registration is rejected with conflict (Device A is still active).

#### Step 2: Wait for Stale Takeover Threshold

If Device A has not been seen for >14 days, the stale takeover threshold is reached.

#### Step 3: Device B Re-Registers (Stale Takeover)

Device B sends the registration request again.

**Expected outcome**:
1. Registry state transitions to `Active(device_id=Device B)`
2. Old-device pending custody entries are purged
3. Device B is now the active device

### 3.3 Emergency Abandonment (No Waiting)

If immediate abandonment is required (e.g., compromised device), an authorized operator can force abandonment:

```
# This requires operator access to the relay node
# Not available to end users
```

**Expected outcome**:
1. Registry state transitions to `Abandoned(timestamp)`
2. All pending custody records for this identity are purged
3. Any future sends to this identity are rejected with `identity_abandoned`

### 3.4 Verification

```bash
# Check registration state
curl -X GET http://localhost:8080/api/v1/registration/state/<identity_id>

# Expected response after abandonment:
{
    "state": "Abandoned",
    "timestamp": <abandonment timestamp>
}
```

### 3.5 Failure Modes

| Failure | Cause | Recovery |
|---------|-------|----------|
| Stale takeover rejected | Device A seen within 14-day window | Wait for threshold or force abandonment |
| Abandoned identity still accepting sends | Registry state not enforced | Check relay custody enforcement logic |
| Pending records not purged | Purge logic failed | Manual cleanup of custody store |

## 4. Sender-Facing Error Handling

### 4.1 Error Codes

| Error Code | Meaning | Sender Action |
|------------|---------|---------------|
| `identity_device_mismatch` | Intended device does not match active device | Retry with correct device ID or wait for handover |
| `identity_abandoned` | Identity has been abandoned | Contact recipient through alternative channel |
| `delivery_failed` | Transient delivery failure | Retry (automatic) |

### 4.2 UI Behavior

**Android**:
- `DeliveryStateSurface` renders `rejected` state for terminal failures
- `ChatScreen` shows snackbar with error details
- Pending outbox stops retrying terminal failures

**iOS**:
- `MainTabView` renders `rejected` state in delivery legend
- Pending outbox stops retrying terminal failures

## 5. Testing Procedures

### 5.1 Handover Test

1. Install app on Device A, create identity
2. Install app on Device B, restore same identity
3. Device B attempts registration → expect conflict
4. Device A initiates handover to Device B
5. Device B re-registers → expect success
6. Send message to identity → expect delivery to Device B

### 5.2 Abandonment Test

1. Install app on Device A, create identity
2. Install app on Device B, restore same identity
3. Device B attempts registration → expect conflict
4. Simulate 14-day stale threshold (or force abandonment)
5. Device B re-registers → expect success (stale takeover)
6. Send message to identity → expect delivery to Device B

### 5.3 Rejection Test

1. Install app on Device A, create identity
2. Install app on Device B, restore same identity
3. Device A initiates abandonment (no target)
4. Send message to identity → expect `identity_abandoned` rejection
5. Verify sender sees `rejected` delivery state in UI

## 6. Monitoring & Observability

### 6.1 Key Metrics

- `relay_registration_total` - Total registration requests
- `relay_registration_rejected_total` - Rejected registrations (by reason)
- `relay_handover_total` - Clean handovers completed
- `relay_abandonment_total` - Identity abandonments
- `relay_custody_rejected_total` - Custody rejections (by reason)

### 6.2 Log Patterns

```
# Successful handover
INFO: Handover initiated: identity=<id> from=<device_a> to=<device_b>
INFO: Handover completed: identity=<id> active_device=<device_b>

# Abandonment
INFO: Identity abandoned: identity=<id> timestamp=<ts>
WARN: Custody rejected: identity=<id> reason=abandoned

# Stale takeover
INFO: Stale takeover: identity=<id> old_device=<device_a> new_device=<device_b> stale_days=<days>
```
