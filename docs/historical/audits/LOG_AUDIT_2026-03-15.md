## Android Log Analysis (android_live.log)

### Issue 1: `msg=unknown` Message ID Propagation Failure
**MASTER_BUG_TRACKER Reference:** AND-DELIVERY-001

**Log Evidence:**
```
Line 15: delivery_attempt msg=unknown medium=core phase=direct outcome=success detail=ctx=send route=... latency=464ms
Line 24: delivery_attempt msg=unknown medium=core phase=direct outcome=failed detail=ctx=send route=... reason=Network error
Line 46: delivery_attempt msg=unknown medium=core phase=direct outcome=failed detail=ctx=outbox_retry route=... reason=Network error
Line 81: delivery_attempt msg=unknown medium=relay-circuit phase=retry outcome=attempt detail=ctx=send route=... relays=2
Line 92: delivery_attempt msg=unknown medium=relay-circuit phase=retry outcome=success detail=ctx=send route=... latency=366ms
```

**Root Cause Analysis:**
The `msg=unknown` appears in delivery_attempt logs when the message UUID is not being propagated to the logging layer. This occurs because:
1. The `sendMessage()` function passes message content but the correlation ID is lost in the async transport layer
2. The `delivery_attempt` log macro receives `null` or empty message_id from the Rust FFI boundary

**Fix Recommendation:**
- **LOC Estimate:** ~25 LOC in MeshRepository.kt
- **Changes Required:**
  1. Capture message UUID BEFORE async send call
  2. Pass message_id explicitly to delivery_attempt logging
  3. Add defensive null check: `msg=${messageId ?: "UNKNOWN_CORRELATION"}`

### Issue 2: Core Transport Network Error (Repeated)
**MASTER_BUG_TRACKER Reference:** CROSS-RELAY-001, AND-CELLULAR-001

**Log Evidence:**
```
Line 23: Core-routed delivery failed for 12D3KooW...: Network error; trying alternative transports
Line 44: Core-routed delivery failed for 12D3KooW...: Network error; trying alternative transports
Line 94: Relay-circuit retry failed for 12D3KooW...: Network error
```

**Root Cause Analysis:**
The core TCP transport is failing with "Network error" which maps to `IronCoreError.NetworkError` (error code 4). This indicates:
1. TCP connection to relay servers timing out or being refused
2. Possible NAT traversal failure on the Android device
3. Firewall or carrier-level blocking of outbound TCP to relay ports (9001, 9010)

**Fix Recommendation:**
- **LOC Estimate:** ~40 LOC across 2 files
- **Changes Required:**
  1. Add connection timeout diagnostic logging before attempting dial (~15 LOC)
  2. Implement aggressive relay bootstrap retry with exponential backoff (~15 LOC)
  3. Add fallback to QUIC/UDP when TCP fails (per CELLULAR_NAT_SOLUTION.md) (~10 LOC)

**Log Evidence:**
```

```

**Pattern Analysis:**
- Error repeats approximately **every 1 second** (timestamps show 01:50:45, 01:50:46, 01:50:47, etc.)
- Same peer ID (`12D3KooWCitizuothXVuqeGShBywNtQ2H8j3sxFap8TLHSYpounv`) being retried
- iOS is stuck in aggressive retry loop without backoff

**Root Cause:**
1. iOS retry logic has no exponential backoff - fires every ~1 second
2. The relay circuit dial (Line 6, 55) shows it's attempting `/ip4/34.135.34.73/tcp/9001/p2p-circuit/` but failing
3. No circuit breaker pattern - continues retrying indefinitely

**Fix Recommendation (✅ IMPLEMENTED):**
- **LOC Estimate:** ~80 LOC across 1 file (MeshRepository.swift)
- **Implementation Location:** Lines 116-126 (state), Lines 4005-4086 (logic)
- **Changes Implemented:**
  1. Added exponential backoff: 1s → 2s → 4s → 8s → 16s → 32s (cap)
  2. Added circuit breaker: pause after 10 consecutive failures for 5 minutes
  3. Added state tracking: `consecutiveDeliveryFailures`, `lastFailureTime`
  4. Added backoff check before relay-circuit attempt
  5. Reset failure count on successful delivery
