## v0.2.1 Planning: WS13 & WS14

### Scope Definition
Per `docs/MILESTONE_PLAN_V0.2.0_ALPHA.md`:
- **WS13 and WS14 are planned v0.2.1 follow-up work**
- They are **NOT** unfinished v0.2.0 requirements
- They are **NEW FEATURES** for next release

### WS13: Single Active Device (Tight Pairing)
**Status**: Planned, not started
**Document**: `docs/V0.2.1_SINGLE_ACTIVE_DEVICE_TIGHT_PAIR_PLAN.md`

**Objective**: Enforce strict destination pairing to prevent:
- Multi-device active collisions
- Stale/recycled identity routing
- Ambiguous relay custody ownership

**Core Changes**:
1. Add persisted `device_id` (UUIDv4) to identity
2. Add `seniority_timestamp` for installation precedence
3. Update contact routing to include `last_known_device_id`
4. Extend relay protocol with device targeting
5. Implement device collision detection
6. Add migration/activation flows

**Phases**: WS13.1 through WS13.6

**Estimated Effort**: 15-25 hours (cross-cutting protocol change)

## Recommendation: Complete v0.2.0 First

### Why Not Start WS13/WS14 Now?

**From Planning Docs**:
> "Current v0.2.0 focus is reliability hardening + parity + app-store alpha readiness; introducing this now materially increases regression risk."

**Reasons**:
1. **v0.2.0 is 95%+ complete** - just needs validation
2. **WS13 is cross-cutting** - touches identity, contacts, transport, relay, UI, migration
3. **Risk of regression** - could break newly-fixed features
4. **Clean milestone boundary** - ship v0.2.0, then start v0.2.1

### Proper Sequence

**Phase 1: v0.2.0 Closeout** (2-4 hours)
1. Deploy to test devices
2. Validate all fixes work
3. Run 24-hour soak test
4. Document any remaining issues
5. Create v0.2.0 release tag

**Phase 2: v0.2.1 Kickoff** (after v0.2.0 ships)
1. Review WS13 plan thoroughly
2. Create WS13.1 implementation branch
3. Begin device_id persistence work
4. Iterative development with testing
