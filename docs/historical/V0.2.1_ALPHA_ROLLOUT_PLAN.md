## 4. Known Open Risks at Rollout

These are **accepted** for alpha launch. None are blocking, but each should be resolved before beta.

### 4.1 HIGH Priority (fix in v0.2.2)

| Risk ID | Description | Impact | Mitigation |
|---------|-------------|--------|------------|
| **R-WS13.6-01** | Contact duplication during peer discovery | Duplicate UI entries, data inconsistency | Idempotent upsert + unique constraint needed |
| **R-2026-03-14-01** | Android auto backup restores stale data | Privacy concern on fresh install | Set `android:allowBackup="false"` or update `backup_rules.xml` |
| **DEBUG-#4** | Android permission request loop on startup | 9+ requests in 700ms, UX degradation | Add deduplication flag + 500ms debounce |

### 4.2 MEDIUM Priority (fix in v0.2.2 or v0.2.3)

| Risk ID | Description | Impact | Mitigation |
|---------|-------------|--------|------------|
| **R-WS12-29-01** | iOS send-path `SIGTRAP` crash in BLE manager | Field-visible crash during send/flush | Deploy latest binary, verify no new reports |
| **R-WS12-29-02** | Android stale route/BLE target churn | Delivery convergence delays | Route-candidate ordering mitigations already applied |
| **R-WS14.1-01** | DM-request intent not on wire (payload not versioned) | Unknown-sender inference only | Keep as compatibility baseline; defer versioned payload |
| **R-WS14.2-01** | iOS DM Request inbox derived from contact notes | Fragile note-based state | Re-evaluate dedicated request store in WS14.5 |
| **DEBUG-#3** | Discovered peers persist 6s after discovery stops | Minor UI lag | Explicit cleanup on stop |

### 4.3 LOW Priority (governance/infrastructure)

| Risk ID | Description | Impact | Mitigation |
|---------|-------------|--------|------------|
| **R-WS13-CF-01** | GitHub CI `action_required` signal quality | PR CI status unreliable | Local verification remains authoritative |
| **R-WS13-CF-02** | iOS/Docker validation environment-gated | Needs macOS for xcodebuild | Carry-forward; separate from code readiness |
| **R-WS13.5-02** | iOS device launch not yet verified (device locked) | Physical proof gap | Unlock device and rerun `install-device.sh` |
| **R-WS14-AUTO-01** | Hourly automation drift risk | Stale branch execution | WS14 handoff/prompt docs now authoritative |

## 5. Rollout Instructions

### 5.1 Android Rollout

**Step 1: Build release APK**
```bash
# 1. Cross-compile Rust core for Android targets
rustup target add aarch64-linux-android armv7-linux-androideabi x86_64-linux-android
export ANDROID_NDK_HOME=/path/to/ndk/26.1.10909125

cargo build --target aarch64-linux-android --release
cargo build --target armv7-linux-androideabi --release
cargo build --target x86_64-linux-android --release

# 2. Copy JNI libraries
mkdir -p core/target/android-libs/{arm64-v8a,armeabi-v7a,x86_64}
cp target/aarch64-linux-android/release/libscmessenger_core.so core/target/android-libs/arm64-v8a/
cp target/armv7-linux-androideabi/release/libscmessenger_core.so core/target/android-libs/armeabi-v7a/
cp target/x86_64-linux-android/release/libscmessenger_core.so core/target/android-libs/x86_64/

# 3. Generate UniFFI Kotlin bindings
cargo run --bin uniffi-bindgen generate core/src/api.udl --language kotlin \
  --out-dir core/target/generated-sources/uniffi/kotlin

# 4. Build signed APK
cd android
./gradlew assembleRelease

# Output: app/build/outputs/apk/release/app-release.apk
```

**Step 2: Install on device**
```bash
# Clean install (recommended for alpha)
android/install-clean.sh

# Or manual ADB install
adb install -r android/app/build/outputs/apk/release/app-release.apk
```

**Step 3: Distribute to testers**
- Upload signed APK to internal distribution channel (Firebase App Distribution, direct APK share, or internal storage).
- Include release notes (see Section 7).
- Minimum Android version: API 26 (Android 8.0 Oreo).

### 5.2 iOS Rollout

**Step 1: Build and sign**
```bash
# 1. Cross-compile Rust core for iOS
rustup target add aarch64-apple-ios aarch64-apple-ios-sim

cargo build --target aarch64-apple-ios --release
cargo build --target aarch64-apple-ios-sim --release

# 2. Create XCFramework
xcodebuild -create-xcframework \
  -library target/aarch64-apple-ios/release/libscmessenger_core.a -headers core/include/ \
  -library target/aarch64-apple-ios-sim/release/libscmessenger_core.a -headers core/include/ \
  -output SCMessengerCore.xcframework

# 3. Copy bindings and framework to iOS project
iOS/copy-bindings.sh

# 4. Build and install on device (auto-detects Team ID and UDID)
./iOS/install-device.sh
```

**Step 2: Distribute to testers**
- Use TestFlight for broader alpha distribution:
  1. Archive the app in Xcode (Product → Archive)
  2. Upload to App Store Connect
  3. Add internal/external testers in TestFlight
- Or use direct device install via `install-device.sh` for small cohorts.
- Minimum iOS version: 17.0.

### 5.3 Relay Node Deployment

A relay node is required for internet-based message delivery between devices.

```bash
# Deploy headless relay to GCP (or any Linux server)
./scripts/deploy_gcp_node.sh

# Verify relay functionality
./scripts/test_gcp_node.sh
```

## 7. Release Notes (v0.2.1-alpha)

### New Features
- **Contact Blocking with Evidentiary Retention**: Block a contact to hide their messages from your inbox while securely retaining them in the database. Unblocking restores all hidden messages.
- **Contact Block + Delete (Cascade Purge)**: Permanently block and delete a contact — all their messages are purged and future messages are rejected at the network layer.
- **Single Active Device Enforcement (WS13)**: Your identity can only be active on one device at a time. Pairing a new device automatically revokes the previous session.
- **DM Notifications (WS14)**: Receive push notifications for direct messages. Notifications are automatically suppressed for blocked contacts.

### Bug Fixes
- Fixed Android startup crash caused by `IronCoreException.NotInitialized` during history sync before identity readiness.
- Improved peer ID matching with case-insensitive comparison across all history operations.
- Fixed blocked+deleted ingress to return explicit error (prevents accidental content exposure).

### Known Issues
- Contact duplication may occur during peer discovery (workaround: manually remove duplicate).
- Android auto backup may restore stale data on fresh install.
- iOS BLE send-path crash reported on some devices (under investigation).

## 8. Rollback Plan

If a critical issue is discovered post-rollout:

1. **Android**: Distribute previous APK (v0.2.0, versionCode 6) via the same channel.
2. **iOS**: If using TestFlight, expire the build and testers auto-revert. For direct install, push previous IPA via `install-device.sh`.
3. **Relay nodes**: Relay nodes are backward-compatible. No relay rollback needed unless relay protocol changed.
4. **Data migration**: v0.2.1 adds `hidden` and `is_deleted` fields with `#[serde(default)]` — existing data deserializes safely. Rollback to v0.2.0 is non-destructive (new fields are ignored by older versions).
