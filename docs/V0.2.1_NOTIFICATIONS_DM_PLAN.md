# WS14 (v0.2.1): Direct Message Notifications + Direct Message Request Notifications (iOS/Android/WASM)

## Summary
Add a new v0.2.1 workstream (`WS14`) to ship notification parity for:
1. Direct Messages
2. Direct Message Requests

across iOS, Android, and WASM, with a **hybrid delivery strategy**:
- Ship full **local notifications now**.
- Add **remote-push interfaces/contracts now** (token + routing abstractions), but defer APNs/FCM/Web Push backend execution.

This plan is decision-complete and aligned to existing repo structure and current v0.2.x planning docs.

---

## Locked Decisions (from this chat)
1. **DM Request source: Both**
   - Infer request for unknown sender (not in contacts).
   - Support explicit request flag/type for forward compatibility.
2. **Delivery mode: Hybrid**
   - Local notifications shipped in v0.2.1.
   - Remote push interfaces prepared in v0.2.1, backend dispatch deferred.
3. **DM Request notification UX**
   - If existing conversation: open that exact conversation.
   - If new sender/request: open Requests Inbox.

---

## Scope

### In scope (WS14)
1. Notification classification policy for `DM` vs `DM Request` across all clients.
2. Notification settings parity (enable/disable notifications, DM Request visibility behavior).
3. iOS local notifications completion (existing `NotificationManager.swift` to production-ready).
4. Android local notifications completion (existing `NotificationHelper.kt` path to production-ready).
5. WASM browser notification path completion (Page Notification API + service-worker bridge wiring).
6. Requests Inbox routing behavior for request notifications.
7. Remote-push interface contracts (token registration, endpoint schema, wake-intent envelope shape) without backend delivery.

### Out of scope (WS14)
1. APNs/FCM/Web Push backend and relay dispatch implementation.
2. Rich media notifications.
3. Group chat notifications.
4. Notification-driven message send/reply actions beyond minimal tap routing unless already stable on a platform.
5. New anti-abuse policy design (consume existing WS10 guardrails only).

## Public Interfaces / Types / API Changes

## 1) Core classification contract (new)
Add a cross-platform classification helper in core to avoid adapter drift:

- `NotificationKind` enum:
  - `DirectMessage`
  - `DirectMessageRequest`
  - `None`

- `NotificationDecision` struct:
  - `kind: NotificationKind`
  - `conversation_id: String`
  - `sender_peer_id: String`
  - `message_id: String`
  - `should_alert: bool`
  - `suppression_reason: Option<String>`

- Function:
  - `classify_notification(inbound_message, contact_state, ui_state, settings) -> NotificationDecision`

Rule order:
1. If notifications disabled -> `None`.
2. If message is self-originated / duplicate / already seen -> `None`.
3. If sender is known contact OR existing conversation exists -> `DirectMessage`.
4. Else unknown sender -> `DirectMessageRequest`.
5. If explicit request flag exists -> force `DirectMessageRequest` unless sender already blocked.

## 2) Message metadata extension (forward-compatible)
Add optional inbound metadata field:
- `is_dm_request: Option<bool>` (default false)

Compatibility:
- If absent, inference still works via unknown-sender rule.
- If present true, classify as request even for first-contact path.

## 3) Notification settings contract (parity)
Add/standardize settings fields across adapters:
- `notifications_enabled: bool` (existing concept normalized)
- `notify_dm_enabled: bool` (default true)
- `notify_dm_request_enabled: bool` (default true)
- `notify_dm_in_foreground: bool` (default false)
- `notify_dm_request_in_foreground: bool` (default true)
- `sound_enabled: bool` (default true)
- `badge_enabled: bool` (default true)

Implementation note (2026-03-14 HST):
- The original single `notify_foreground` draft field was split into per-kind foreground toggles during WS14.1 implementation so the shared contract stays explicit and adapters do not need to infer different defaults from one boolean.

## 4) Routing contract
Define shared deep-link target schema:
- `scm://chat/<peer_id>?message_id=<id>`
- `scm://requests?sender=<peer_id>&message_id=<id>`

Behavior:
- Existing conversation -> route to chat deep link.
- New request -> route to requests deep link.

## 5) Hybrid remote-push interfaces (no backend dispatch yet)
Define cross-platform endpoint contract:
- `NotificationEndpoint`:
  - `platform: ios|android|web`
  - `token_or_subscription: encrypted blob`
  - `capabilities: {dm: bool, dm_request: bool}`
  - `last_seen_ts`
  - `device_id` (align with WS13 model)
- Core/adapters expose:
  - `register_notification_endpoint(...)`
  - `unregister_notification_endpoint(...)`
  - `list_notification_endpoints()`

No APNs/FCM/Web Push send implementation in WS14; contract only.

## Behavior Spec (Decision Complete)

## Notification trigger policy
1. Trigger only on **new inbound** messages.
2. Never notify for:
   - sent/self messages
   - duplicates
   - already-open active thread when `notify_foreground=false` (except DM Requests if configured true)

## DM vs DM Request classification
1. If `is_dm_request=true` -> DM Request.
2. Else if sender exists in contacts OR conversation exists -> DM.
3. Else -> DM Request.

## Tap routing
1. DM notification:
   - Open conversation for sender.
2. DM Request notification:
   - If conversation already exists -> open conversation.
   - Else -> open Requests Inbox.
3. If route target missing due stale state:
   - fallback to Requests Inbox for unknown sender,
   - fallback to conversations list for known sender.

## Dedup and collapse
1. Use `message_id` as notification id key.
2. Collapse by `conversation_id` for DM.
3. Collapse by `sender_peer_id` for DM Request.

## Settings defaults
1. Notifications enabled = true
2. DM enabled = true
3. DM Request enabled = true
4. Foreground notify:
   - DM = false via `notify_dm_in_foreground`
   - DM Request = true via `notify_dm_request_in_foreground`
5. Sound and badge enabled = true

## Residual Risks to pre-register for WS14
1. OS-level throttling/background limitations can suppress delivery-time notifications.
2. Browser notification permissions vary heavily by user agent.
3. Without remote backend, terminated-app scenarios rely on local runtime availability.
4. Request classification could still mislabel rare edge states during migration windows.

Each risk must be logged in the active residual register with acceptance/defer rationale.
