## 1. Compatibility Mode Policy

### 1.1 Legacy No-Device Traffic (Pre-WS13 Clients)

**Policy: Compatibility-accepted with logging**

Pre-WS13 clients send relay requests without `recipient_identity_id` or `intended_device_id`. These requests are accepted under the following conditions:

| Condition | Behavior | Rationale |
|-----------|----------|-----------|
| No `intended_device_id` in request | Accept and route to identity's active device (if any) | Backward compatibility for v0.2.0 clients |
| Identity has no registered active device | Accept and deliver (legacy behavior) | Allows gradual migration |
| Identity has active device registered | Accept and route to active device | Best-effort delivery to known device |
| Identity is in `Abandoned` state | Reject with `identity_abandoned` | Security: prevent delivery to abandoned identities |

**Logging**: All legacy no-device requests are logged with `legacy_no_device=true` for migration tracking.

### 1.2 WS13+ Clients (With Device Metadata)

**Policy: Strict enforcement**

WS13+ clients include `recipient_identity_id` and `intended_device_id` in relay requests. These requests are strictly enforced:

| Condition | Behavior | Rationale |
|-----------|----------|-----------|
| `intended_device_id` matches active device | Accept | Correct destination |
| `intended_device_id` mismatches active device | Reject with `identity_device_mismatch` | Prevent delivery to wrong device |
| Identity is in `Handover(target)` state and `intended_device_id` matches target | Accept and redirect | Support clean handover |
| Identity is in `Abandoned` state | Reject with `identity_abandoned` | Security: prevent delivery to abandoned identities |

## 2. Migration Matrix: Pre-WS13 → WS13

### 2.1 Data Migration

| Component | Pre-WS13 State | WS13 State | Migration Path |
|-----------|----------------|------------|----------------|
| Identity | `public_key` only | `public_key` + `device_id` + `seniority_timestamp` | Auto-generated on first WS13 startup; existing identities receive generated `device_id` without key rotation |
| Contacts | `peer_id`, `public_key`, `nickname`, `notes` | + `last_known_device_id` | `None` default for legacy contacts; populated on next interaction |
| Relay Requests | `(peer_id, data)` | + `recipient_identity_id`, `intended_device_id` | `None, None` for legacy senders; `#[serde(default)]` ensures backward compatibility |
| Relay Registry | Not present | `RelayRegistry` keyed by `identity_id` | Empty registry on upgrade; populated on first registration |

### 2.2 Upgrade Scenarios

| Scenario | Expected Behavior | Verification |
|----------|-------------------|--------------|
| Fresh install (WS13) | New `device_id` generated, empty registry | `device_id` is valid UUIDv4, persists across restarts |
| Upgrade from v0.2.0 | Existing identity preserved, new `device_id` generated | Identity key unchanged, `device_id` present after upgrade |
| Restore from backup | Identity key restored, new `device_id` generated for this installation | Backup payload is portable identity only; device metadata is installation-local |
| Mixed mesh (v0.2.0 + WS13) | Legacy requests accepted in compatibility mode | v0.2.0 clients can send/receive; WS13 clients enforce tight-pair |

### 2.3 Breaking Changes

**None for v0.2.0 clients**: All WS13 changes are additive. Pre-WS13 clients continue to operate without modification.

**For WS13+ clients**: Strict enforcement is opt-in via `intended_device_id` presence. Clients that don't send device hints continue in compatibility mode.

## 3. Enforcement Mode Transition

### 3.1 Phase A: Compatibility Mode (Current)

- Legacy no-device requests accepted
- WS13+ requests strictly enforced
- All transitions logged for migration tracking

### 3.2 Phase B: Enforced Mode (Future)

- Strict enforcement for all clients advertising WS13 support
- Legacy clients still accepted but logged as deprecated
- Migration warnings surfaced to operators

### 3.3 Phase C: Full Policy (Future)

- All requests require `intended_device_id`
- Legacy no-device requests rejected
- Requires client upgrade penetration >95%

**Decision point**: Phase B/C transition is deferred until after v0.2.1 release and sufficient client upgrade penetration.
