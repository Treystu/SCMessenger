## WS13.2

### R-WS13.2-01 - Tight-pair enforcement is blocked by the current swarm send boundary carrying only `(peer_id, envelope_data)`

- Phase: WS13.2
- Status: **CLOSED** — resolved in this session
- Resolution: Transport/API boundary widened. `SwarmCommand::SendMessage`, `SwarmHandle::send_message`, `SwarmBridge::send_message`, `RelayRequest`, and `Contact` now carry `recipient_identity_id: Option<String>` and `intended_device_id: Option<String>`. `api.udl` updated. All existing callers receive `None, None` for backward compatibility. `#[serde(default)]` on `RelayRequest` ensures pre-WS13 relay nodes deserialize cleanly.

### R-WS13.2-02 - Required Android/iOS verification tooling is unavailable on this host for adapter-surface changes

- Phase: WS13.2
- Status: **Open** (residual — tooling environment, not code correctness)
- Impact: Mobile Kotlin/Swift consumers generated from `api.udl` must regenerate their bindings and call-sites with the new signature `void send_message(string peer_id, bytes data, string? recipient_identity_id, string? intended_device_id)`. The Rust core and UniFFI UDL are correct; the generated adapter call-sites are not verifiable on this Linux host.
- Evidence:
  - `command -v xcodebuild` returned no result on this Linux host.
  - `command -v cargo-ndk` returned no result.
  - `ANDROID_HOME` is unset and `rustup target list --installed` shows only `x86_64-unknown-linux-gnu`.
- Next action:
  - Re-run binding generation on a host with Android SDK/NDK plus macOS/iOS build tooling to produce and verify the updated Swift/Kotlin call-sites.
  - Alternatively, provide CI/build evidence (iOS build + Android Gradle build) that satisfies the mandatory adapter verification policy before claiming WS13.2 fully platform-verified.

- Phase: WS13.4
- Status: Open
- Impact: The authoritative relay registry now exists and rejects mismatched/abandoned device-targeted relay requests, but pending custody records are not yet migrated on clean handover or purged on abandonment, and sender-facing rejection UX has not been plumbed through clients.
