## WS13.5

### R-WS13.5-01 - Fresh-install Android startup can hit non-fatal history-sync ordering before identity/core readiness settles

- Phase: WS13.5
- Status: **Closed**
- Impact: On a clean Android install, startup can emit `IronCoreException.NotInitialized` from history-sync preparation before the local identity/core is ready. No fatal crash was observed on the March 13, 2026 Pixel 6a verification run, but the startup ordering still produces noisy false-positive error logs and could hide a real regression later.
- Evidence:
  - Physical-device launch on Pixel 6a succeeded and `com.scmessenger.android/.ui.MainActivity` reached the resumed foreground state.
  - `adb logcat` captured non-fatal `MeshRepository$sendHistorySyncIfNeeded: uniffi.api.IronCoreException$NotInitialized: Not initialized` during fresh startup.
- Resolution (2026-03-30):
  - `android/app/src/main/java/com/scmessenger/android/data/MeshRepository.kt`: `sendHistorySyncIfNeeded()` now gates on identity readiness by calling `core.getIdentityInfo().identityId != null` before attempting `prepareMessageWithId()`. Catches only `IronCoreException.NotInitialized` for the guard check with broader-exception fallback and a warning log. This prevents the noisy false-positive logs without masking real errors.

## Contact Block/Delete State Machine (v0.2.1 rollout)

### R-BLOCK-01 - Contact Block/Unblock evidentiary retention and cascade delete are now implemented

- Phase: v0.2.1 rollout blocker
- Status: **Closed**
- Impact: Previously the block state machine lacked evidentiary retention (messages from blocked peers were not persisted) and had no cascade purge for the blocked+deleted state.
- Resolution (2026-03-30):
  - `core/src/store/blocked.rs`: Added `is_deleted: bool` to `BlockedIdentity`. Added `block_and_delete()`, `is_blocked_and_deleted()`, and `blocked_only_peer_ids()` to `BlockedManager`.
  - `core/src/store/history.rs`: Added `hidden: bool` to `MessageRecord`. Refactored `recent()` to filter hidden records. Added `recent_including_hidden()` for evidentiary/admin access. Added `unhide_messages_for_peer()`.
  - `core/src/mobile_bridge.rs`: Same `hidden` field and filtering in the mobile bridge `HistoryManager`. Added `unhide_messages_for_peer()`.
  - `core/src/lib.rs`: `receive_message()` now checks blocked state at ingress — blocked+deleted drops the payload; blocked-only stores with `hidden: true` and suppresses the delegate callback. `unblock_peer()` restores hidden messages. New `block_and_delete_peer()` sets the flag and purges history.
  - `core/src/api.udl`: Exposed `hidden: boolean` on `MessageRecord` and `block_and_delete_peer()` on `IronCore`.
  - `core/tests/integration_contact_block.rs`: Three integration tests verify all three scenarios (hidden retention, unblock restores, cascade purge + ingress drop). All pass.
  - `cargo test --workspace` 670 pre-existing tests + 3 new tests, 0 failures.

### R-WASM-01 — WASM `uniffi_core` `Send` bound resolved via feature gating

- Phase: CI / Build Infrastructure
- Status: **Resolved** (2026-03-31)
- Impact: WASM CI builds for `wasm32-unknown-unknown` previously failed because `uniffi_core` 0.31's `UniffiCompatibleFuture` trait required `Send`, which is not available on wasm32 single-threaded targets.
- Root cause: The `wasm-unstable-single-threaded` feature (which relaxes the `Send` bound) was only enabled in the `scmessenger-wasm` crate's dependency on `uniffi`, but the CI also ran `cargo check -p scmessenger-core --target wasm32-unknown-unknown` which compiled core WITHOUT the feature.
- Fix applied:
  1. Added `wasm` feature to `core/Cargo.toml` that forwards to `uniffi/wasm-unstable-single-threaded`
  2. Updated `wasm/Cargo.toml` to enable `scmessenger-core/wasm` feature
  3. Updated `.github/workflows/ci.yml` to pass `--features wasm` when checking core for wasm32
- Verification: Local `cargo build --workspace` passes; `cargo test --workspace` 670+ pass, 0 fail
- Future consideration: When `uniffi_core` upstream resolves wasm32 `Send` handling natively (tracked in [mozilla/uniffi-rs#2234](https://github.com/mozilla/uniffi-rs/issues/2234)), the `wasm` feature can be removed
