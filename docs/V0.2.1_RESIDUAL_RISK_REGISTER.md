## WS13.2

### R-WS13.2-01 - Tight-pair enforcement is blocked by the current swarm send boundary carrying only `(peer_id, envelope_data)`

- Phase: WS13.2 preflight
- Status: Open
- Impact: The relay path cannot carry recipient identity or `intended_device_id` today because `SwarmCommand::SendMessage` / `SwarmBridge::send_message` expose only `peer_id` plus opaque encrypted bytes. Without widening that boundary, WS13.2 request metadata plumbing and WS13.4 custody enforcement cannot be implemented correctly against encrypted envelopes.
- Evidence:
  - `core/src/transport/swarm.rs` still defines `SwarmCommand::SendMessage { peer_id, envelope_data, ... }`.
  - `core/src/api.udl` still exposes `SwarmBridge.send_message(string peer_id, bytes data)`.
  - `core/src/transport/behaviour.rs` still defines relay requests with only destination peer bytes, message bytes, and message ID.
- Next action:
  - Introduce a compatibility-safe transport API widening that carries recipient identity and optional `intended_device_id` alongside encrypted bytes.
  - Re-audit Android/iOS/Web binding impacts before editing adapter surfaces.

- Phase: WS13.2 preflight
- Status: Open
- Impact: Repo policy requires platform verification when adapter/binding surfaces change. This host cannot currently provide trustworthy Android/iOS verification for the API widening that WS13.2+ requires.

  - `command -v cargo-ndk` returned no result.
