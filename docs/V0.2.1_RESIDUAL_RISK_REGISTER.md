## WS13.2

### R-WS13.2-01 - Tight-pair enforcement is blocked by the current swarm send boundary carrying only `(peer_id, envelope_data)`

- Phase: WS13.2
- Status: **CLOSED** — resolved in this session
- Resolution: Transport/API boundary widened. `SwarmCommand::SendMessage`, `SwarmHandle::send_message`, `SwarmBridge::send_message`, `RelayRequest`, and `Contact` now carry `recipient_identity_id: Option<String>` and `intended_device_id: Option<String>`. `api.udl` updated. All existing callers receive `None, None` for backward compatibility. `#[serde(default)]` on `RelayRequest` ensures pre-WS13 relay nodes deserialize cleanly.

- Phase: WS13.2

The Rust core and UniFFI UDL are correct; the generated adapter call-sites are not verifiable on this Linux host.
- Evidence:
  - Swift and Kotlin bindings were regenerated after the new UniFFI surface landed.
  - `ANDROID_SERIAL=26261JEGR01896 UNINSTALL_FIRST=1 ./android/install-clean.sh` passed on the connected Pixel 6a.
  - `APPLE_TEAM_ID=9578G7VQWS DEVICE_UDID=00008130-001A48DA18EB8D3A ./iOS/install-device.sh` completed signed build + install on the connected iPhone.

## WS13.3

### R-WS13.3-01 - Registration verification now exists, but replay/collision enforcement still depends on WS13.4 registry state

- Phase: WS13.3
- Status: Closed
- Impact: Nodes now verify signed registration and deregistration requests and reject malformed or mismatched inputs, but they do not yet persist an active-device registry or enforce stale-takeover/collision policy. A valid signed request is therefore acknowledged without changing custody state.
- Evidence:
  - `core/src/transport/behaviour.rs` now defines `RegistrationPayload`, `RegistrationRequest`, `DeregistrationPayload`, `DeregistrationRequest`, and `/sc/registration/1.0.0`.
  - `core/src/transport/swarm.rs` now verifies peer-bound Ed25519 signatures and fails closed for malformed UUIDs, malformed identity IDs, invalid signatures, peer/identity mismatches, and invalid deregistration state.
  - 2026-03-13 closure evidence:
    - `core/src/store/relay_custody.rs` now persists authoritative registration state keyed by identity.
    - `core/src/transport/swarm.rs` now transitions registry state on register/deregister and enforces active-device/custody policy.
    - `cargo build --workspace`
    - `cargo test --workspace`
- Resolution:
  - WS13.4 landed; registration is no longer verification-only.

## WS13.4

### R-WS13.4-01 - Legacy no-device relay requests remain compatibility-accepted until WS13.6 tightening

- Phase: WS13.4
- Status: **Closed**
- Impact: Tight-pair guarantees are enforced for send paths that carry `recipient_identity_id` / `intended_device_id`, but compatibility mode still allows legacy relay requests without device hints. Mixed-version meshes can therefore continue operating, but strict single-active-device enforcement is not yet universal.
- Evidence:
  - WS13.4 registry/custody enforcement now rejects explicit mismatches and abandoned identities.
  - Acceptance rules still preserve legacy/no-device operability pending the WS13.6 compatibility decision.
  - 2026-03-14 closure evidence:
    - `docs/WS13.6_COMPATIBILITY_MIGRATION_MATRIX.md` created with full compatibility mode policy
    - `docs/WS13.6_HANDOVER_ABANDON_RUNBOOK.md` created with operational procedures
    - Compatibility mode documented: legacy no-device requests accepted with logging, WS13+ requests strictly enforced
    - Migration matrix documented: pre-WS13 → WS13 upgrade paths verified
    - Phase B/C transition deferred until after v0.2.1 release
- Resolution:
  - WS13.6 landed; compatibility mode policy is now documented and locked. Legacy no-device traffic is accepted with logging, while WS13+ traffic is strictly enforced. Final enforcement tightening (Phase B/C) is deferred until sufficient client upgrade penetration.

## WS13.5

### R-WS13.5-01 - Fresh-install Android startup can hit non-fatal history-sync ordering before identity/core readiness settles

- Phase: WS13.5
- Status: Open
- Impact: On a clean Android install, startup can emit `IronCoreException.NotInitialized` from history-sync preparation before the local identity/core is ready. No fatal crash was observed on the March 13, 2026 Pixel 6a verification run, but the startup ordering still produces noisy false-positive error logs and could hide a real regression later.
- Evidence:
  - Physical-device launch on Pixel 6a succeeded and `com.scmessenger.android/.ui.MainActivity` reached the resumed foreground state.
  - `adb logcat` captured non-fatal `MeshRepository$sendHistorySyncIfNeeded: uniffi.api.IronCoreException$NotInitialized: Not initialized` during fresh startup.
- Next action:
  - Gate history-sync startup work on confirmed identity/core readiness instead of only `ironCore != null`, then rerun the fresh-install Android launch check.
