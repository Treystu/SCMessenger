## Immediate Live Verification Feedback Loop (WS12.30 Execution Mode)

Use the new non-destructive iterative harness copy at `scripts/run5-live-feedback.sh`.
It preserves existing `run5.sh` behavior by wrapping it instead of modifying it.

Primary command per fix-step:

```bash
./scripts/run5-live-feedback.sh --step=<fix-id> --time=5 --attempts=3
```

Recommended strict mode for convergence closures:

```bash
./scripts/run5-live-feedback.sh \
  --step=<fix-id> \
  --time=6 \
  --attempts=3 \
  --require-receipt-gate
```

Sequential phase contract (must pass before next phase):

1. Phase A - Build/Deploy:
   - Deploy Android+iOS (`scripts/deploy_to_device.sh both`) unless explicitly skipped.
2. Phase B - 5-node update/capture:
   - Run `run5.sh` with `--update` to refresh headless nodes and capture synchronized 5-node logs.
3. Phase C - Log-health gate:
   - Require non-empty logs for all 5 nodes (`gcp`, `osx`, `android`, `ios-device`, `ios-sim`).
4. Phase D - Pair-matrix gate:
   - Require full directed visibility matrix (all node pairings present).
5. Phase E - Crash/fatal gate:
   - Fail on iOS crash markers and Android fatal markers in capture window.
6. Phase F - Deterministic verifiers:
   - `verify_relay_flap_regression.sh`
   - `verify_ble_only_pairing.sh`
   - `verify_receipt_convergence.sh` (warning-only unless `--require-receipt-gate` is set).
7. Phase G - Evidence packaging:
   - Persist attempt artifacts to `logs/live-verify/<step>_<timestamp>/attempt_*`.

Iteration rule:

1. If any required gate fails, treat the step as not done.
2. Apply fix refinement.
3. Re-run same `--step=<fix-id>` loop until required gates pass.
4. Only then move to the next issue in execution order.
