# Plan: Update Repo Map with Function Extraction (Revised)

## Overview
Create a comprehensive repository map that includes not only file metadata but also a lightweight index of top‑level function / method names in each source file. This map will serve as a **pre‑compiled knowledge graph** that future LLM agents can query quickly, reducing the need for on‑the‑fly static analysis. It also facilitates automated wiring‑up checks and dependency resolution.

## How the Map Helps LLMs
- **Fast lookup** -- Agents can `grep` the JSON for a function name without parsing the entire codebase. 
- **Documented context** -- Knowing the declaring file and signature allows the LLM to correctly reference functions when generating wiring code. 
- **Consistency checks** -- A missing function in the map triggers a quick diagnostic that the agent can flag as a potential wiring gap. 
- **Cross‑module wiring** -- Relationships between modules can be inferred from the map, enabling automated generation of boilerplate plumbing (e.g., dependency injection). 
- **Regression safety** -- Whenever a change occurs, re‑generating the map alerts the LLM to any new or removed symbols that may affect integration points.

## Implementation Plan
1. **Generate repository artifact** -- `scripts/generate_repos_data.py`
   * Walks the repo, skips ignored directories.
   * For each source file (`.ts`, `.tsx`, `.js`, `.jsx`, `.swift`, `.kt`, `.java`), uses Tree‑Sitter parsers to collect function/method entries.
   * Emits a JSON document with structure:
     ```json
     {
       "files": [
         {
           "path": "src/index.ts",
           "type": "file",
           "size": 1324,
           "modified": "2026-06-18T12:34:56Z",
           "functions": [
             {"name": "initializeApp", "signature": "initializeApp(cfg: Config): void", "doc": "Initialises the app."}
           ]
         }
       ]
     }
     ```
2. **Pre‑compile downstream database** -- Optionally, transform the JSON into a lightweight SQLite DB (`repo_map.db`) with two tables:
   * `files(path TEXT PRIMARY KEY, type TEXT, size INTEGER, modified TEXT)`
   * `functions(id INTEGER PRIMARY KEY, file_path TEXT, name TEXT, signature TEXT, doc TEXT)`
   This DB gives LLM agents O(1) lookup via SQL and can be queried by agents with a simple CLI wrapper.
3. **Documentation** -- `docs/REPO_MAP.md`
   * Schema description.
   * How to regenerate (`scripts/generate_repos_data.py`).
   * How agents can query the map or DB (e.g., `jq -r '...` or a provided Python helper).
4. **CI Integration** -- Add a GitHub Action that re‑generates the map on every push to `main` and updates the JSON/DB in a dedicated branch (`repo-map-update`).
5. **Verification** -- After generation:
   * Run `jq . REPO_MAP.json | head`.
   * Run a quick SQL query against the DB to confirm presence of a known function.
6. **Future Maintainability** -- Document that any structural change (adding/removing modules, renaming functions) should trigger regeneration.
