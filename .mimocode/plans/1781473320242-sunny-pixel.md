# Identity Generation Flow — Comprehensive Fix Plan

## Problem Summary

Android identity generation hangs for ~2 minutes showing "About 4 seconds remaining", then Settings shows no identity and prompts redundant recreation. Identity eventually loads ~5 minutes after app start.

## Root Cause Analysis (4 Cascading Bugs)

**Single root cause**: `ensureServiceInitialized()` in `MeshRepository.kt:4684` has a 10-second timeout. On cold start, the Rust core service (sled DB init, manager initialization, data migration) can take 30-120s. When the timeout expires, `createIdentity()` silently returns at line 4528-4530 without:
- Creating the identity
- Firing any progress callbacks beyond `PreparingStorage`
- Throwing an error
- Notifying the UI

This cascades into all 4 reported symptoms:
1. **Hang**: Progress stuck on stage 1 because callbacks never fire after the silent return
2. **Wrong ETA**: `TOTAL_ETA_MS = 3900L` doesn't account for service startup time
3. **No identity in Settings**: `publishIdentityInfo()` never called, StateFlow stays null
4. **Redundant recreation**: `IdentityScreen` sees `identityInfo?.initialized != true` and shows creation form

## Phase 1: Fix `ensureServiceInitialized()` — The Root Cause

**File**: `android/app/src/main/java/com/scmessenger/android/data/MeshRepository.kt`

### Change 1a: Increase timeout and add progress callback support (line 4684-4704)

Replace `ensureServiceInitialized()` to:
- Increase timeout from 10s → 60s (cold start on low-end devices)
- Accept an optional `IdentityProgressCallback` parameter
- Fire `PreparingStorage` callback periodically during the wait (every 2s) with elapsed time info
- **Throw an exception** on timeout instead of returning `false` — this propagates to the ViewModel's catch block which already sets `_identityError` and `_isReady = false`

```kotlin
private suspend fun ensureServiceInitialized(
    progress: IdentityProgressCallback = { _, _ -> }
): Boolean {
    // Fast path
    if (meshService?.getState() == uniffi.api.ServiceState.RUNNING && ironCore != null) {
        return true
    }
    ensureServiceInitializedDeferred()
    
    val startTime = System.currentTimeMillis()
    val timeoutMs = 60_000L
    var lastProgressMs = 0L
    while (System.currentTimeMillis() - startTime < timeoutMs) {
        if (meshService?.getState() == uniffi.api.ServiceState.RUNNING && ironCore != null) {
            return true
        }
        val elapsed = System.currentTimeMillis() - startTime
        if (elapsed - lastProgressMs >= 2_000L) {
            lastProgressMs = elapsed
            progress(IdentityCreationEvent.PreparingStorage,
                "Starting secure service… (${elapsed / 1000}s)")
        }
        kotlinx.coroutines.delay(200)
    }
    Timber.e("ensureServiceInitialized timed out after ${timeoutMs}ms")
    return false
}
```

### Change 1b: Propagate timeout as error in `createIdentity()` (line 4528-4531)

Change the silent return to throw:
```kotlin
if (!ensureServiceInitialized(progress) || ironCore == null) {
    throw IllegalStateException("Service initialization timed out — identity cannot be created")
}
```

This ensures the ViewModel's `catch` block fires, setting `_identityError` and resetting `_isCreating`.

### Change 1c: Pass progress callback through to `ensureServiceInitialized()` (line 4527-4528)

```kotlin
progress(IdentityCreationEvent.PreparingStorage, "Waking the encrypted key vault…")
if (!ensureServiceInitialized(progress) || ironCore == null) {
```

## Phase 3: Fix State Propagation — Guarantee UI Update

### Change 3a: Ensure `publishIdentityInfo()` is called even on partial success

**File**: `MeshRepository.kt` — `createIdentity()` (line 4570-4574)

After `initializeIdentity()` succeeds (line 4556), immediately publish identity info BEFORE the backup/swarm steps. This way even if `persistIdentityBackup()` or `initializeAndStartSwarm()` fails, the UI knows identity exists.

```kotlin
ironCore?.initializeIdentity()
// Publish immediately so UI knows identity exists
val created = ironCore?.getIdentityInfo()
if (created != null && created.initialized) {
    cacheIdentityFields(created)
    publishIdentityInfo(created)  // Move this UP from line 4574
}
ensureLocalIdentityFederation()
```

### Change 3b: Add timeout recovery in `IdentityViewModel.loadIdentity()`

```kotlin
val backoffMs = longArrayOf(100L, 200L, 400L, 800L, 1_000L, 1_500L)
```

### Change 3c: Add identity backup check to `IdentityScreen` before showing creation form

Before showing `IdentityNotInitializedView`, check if identity backup exists in SharedPreferences. If it does, show a "Restoring identity…" spinner instead of the creation form:

```kotlin
identityInfo == null || identityInfo?.initialized != true -> {
```

Add `isBackupAvailable()` to `IdentityViewModel`:

## Phase 4: Fix Redundant Recreation Guard

### Change 4a: Add duplicate-creation guard to `IdentityNotInitializedView`

**File**: `IdentityScreen.kt` — `IdentityNotInitializedView` (line 161)

The composable already receives `isCreating` and disables the button. But the deeper issue is that `identityInfo` stays null because the StateFlow hasn't propagated. Change 3a (publishing identity immediately after keygen) fixes the root cause. As defense-in-depth, add a `LaunchedEffect` that polls `meshRepository.isIdentityInitialized()` if `identityInfo` stays null for >5s after creation was triggered.

### Change 4b: Add `VerifyingIdentity` shortcut for already-initialized case

**File**: `MeshRepository.kt` — `createIdentity()` line 4536-4544

When identity already exists, instead of just firing `VerifyingIdentity` and returning, also publish the existing identity info so the UI updates:

```kotlin
if (current?.initialized == true) {
    Timber.d("createIdentity: identity already initialized; skipping")
    cacheIdentityFields(current)
    publishIdentityInfo(current)  // ADD THIS
    progress(IdentityCreationEvent.VerifyingIdentity, "Identity already on disk; verifying…")
    return@withContext
}
```

## Phase 5: Performance Optimizations (Rust Core)

### Change 5a: Cache PeerId in `get_identity_info()`

**File**: `core/src/iron_core.rs` — line 930

The `to_libp2p_peer_id()` is called on every `get_identity_info()` call. Cache it in the `IronCore` struct:

```rust
// Add to IronCore struct:
cached_peer_id: Arc<RwLock<Option<String>>>,

// In get_identity_info():
let libp2p_peer_id = {
    let cached = self.cached_peer_id.read();
    if cached.is_some() {
        cached.clone()
    } else {
        drop(cached);
        let pid = keys.and_then(|k| k.to_libp2p_peer_id().ok()).map(|p| p.to_string());
        *self.cached_peer_id.write() = pid.clone();
        pid
    }
};
```

Invalidate cache in `initialize_identity()` and `import_identity_backup()`.

### Change 5b: Remove `println!` diagnostic in production

**File**: `core/src/iron_core.rs` — lines 935, 939

Change `println!("[IDENTITY_DIAG]...")` to `tracing::debug!(...)` to reduce logcat noise.

## Edge Cases Addressed

| Edge Case | Handling |
|-----------|----------|
| Service startup >60s | Exception propagates to VM catch block, user sees error message |
| Concurrent `createIdentity()` calls | Existing `identityCreationMutex` (line 4519) handles this |
| Identity exists but StateFlow is null | Change 3a publishes immediately after keygen |
| Sled corruption/loss | `isIdentityInitialized()` multi-tier check (backup, sentinel, core, DB file) |
| Consent not granted | `grantConsent()` called before `initializeIdentity()` on both platforms |
| Backup write fails | Identity still published to UI; backup retried on next `ensureLocalIdentityFederation()` |
| User navigates to Settings during creation | `IdentityViewModel` observes `meshRepository.identityInfo` StateFlow which gets published |
| Cold start thundering herd | `isRefreshing` AtomicBoolean in MainVM, `identityCreationMutex` in repo |
| iOS missing consent | Change 6a adds `grantConsent()` |
| PeerId derivation slow | Change 5a caches it |

## Verification Strategy

1. **Cold start test**: Clear app data, launch app, create identity. Verify:
   - Progress shows "Starting secure service" with elapsed time (not "4 seconds remaining")
   - Progress advances through all 7 stages
   - Identity appears in Settings after creation
   - No redundant recreation prompt

2. **Warm start test**: App already running, go to Settings → Identity → Create. Verify:
   - Progress skips "InitializingService" (service already running)
   - Shows ~4s ETA (accurate for warm path)

3. **Service timeout test**: Simulate slow service (add artificial delay in `startMeshService`). Verify:
   - Progress shows elapsed time during wait
   - If timeout exceeded, user sees error message (not silent hang)

4. **Concurrent creation test**: Tap "Generate Identity" rapidly. Verify:
   - Only one creation proceeds (mutex guard)
   - Second tap is silently dropped

5. **iOS parity test**: Same cold/warm start tests on iOS. Verify:
   - `grantConsent()` is called
   - `isIdentityInitialized()` checks Keychain backup

6. **Cross-platform identity exchange**: Create identity on Android, export, import on iOS (and vice versa). Verify:
   - Identity resolves correctly
   - PeerId matches
